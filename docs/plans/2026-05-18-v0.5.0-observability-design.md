## 1. 동기 및 결정 제약

### 1.1 문제
v0.4.x 까지 forgen 은 "솔루션 추출 → 매칭 → 주입" 까지는 잘 하지만, **주입된 솔루션이 실제로 쓰였는지** 측정이 산발적입니다. 측정 인프라가 6개 파일(`implicit-feedback.jsonl`, `outcomes/*.jsonl`, `compound-usage.jsonl`, `match-eval-log.jsonl`, 등) 에 분산되어 있어:

- 솔루션별 hit-rate / lifecycle(hot/warm/cold/dead) 분류가 불가능
- `fgx status` 같은 통합 대시보드가 없음 (현 `statusline-cli.ts` 는 1줄 짜리 HUD)
- "이 솔루션 정말 효과 있나" 라는 질문에 답이 없어 자기개선 루프(약한 솔루션 retire, 강한 솔루션 promote) 가 닫히지 않음

### 1.2 결정된 제약 (사용자 선택)
1. **acted_on heuristic**: Multi-signal 처음부터 — prompt keyword + tool pattern + commit diff + compound 추출 4개 가중 합산
2. **Storage**: 기존 v0.4.8 `~/.forgen/sessions.db` (FTS5) 에 `solution_events` 테이블 추가 (단일 DB)
3. **검증 fixture**: 이미 축적된 메모리(`~/.forgen/state/*`, `~/.forgen/me/solutions/*.md`, `~/.claude/projects/.../*.jsonl`) 를 backfill 하여 heuristic 정확도 검증

### 1.3 비목표 (Non-goals)
- LLM 호출 기반 의미 분석 (token cost / latency 폭증). signal 은 모두 결정론적 텍스트/패턴 매칭
- 실시간 dashboard (TUI live refresh) — `--watch` 옵션은 5s polling 으로 충분
- 다중 머신 동기화 — 단일 머신 SQLite 가정

## 3. 데이터 모델 (SQLite schema + 정책)

### 3.1 테이블 (`~/.forgen/sessions.db` 에 추가)

```sql
-- v0.5.0 schema version
CREATE TABLE IF NOT EXISTS schema_version (
  component TEXT PRIMARY KEY,
  version INTEGER NOT NULL,
  applied_at INTEGER NOT NULL
);

CREATE TABLE IF NOT EXISTS solution_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,                    -- epoch ms
  session_id TEXT,                        -- nullable: backfill 시 sid 모를 수 있음
  solution_id TEXT NOT NULL,              -- frontmatter `name` (e.g. '2026-04-10-scored-signal-aggregation')
  event_type TEXT NOT NULL
    CHECK(event_type IN ('matched','surfaced','acted_on')),
  signal_source TEXT,                     -- 'matcher' | 'hook-prepend' | 'skill-invoke' |
                                          -- 'prompt-keyword' | 'tool-pattern' |
                                          -- 'commit-diff' | 'compound-extract' | 'backfill'
  signal_score REAL,                      -- 0.0~1.0; matched=relevance, acted_on=aggregated weighted score
  meta TEXT                               -- JSON blob (matched tags, surface chars, diff sha, etc.)
);

CREATE INDEX IF NOT EXISTS idx_se_solution ON solution_events(solution_id, ts);
CREATE INDEX IF NOT EXISTS idx_se_session  ON solution_events(session_id, ts);
CREATE INDEX IF NOT EXISTS idx_se_type     ON solution_events(event_type, ts);
```

### 3.2 DB 위치 결정 — **단일 파일** (사용자 결정 채택)
- `~/.forgen/sessions.db` 에 통합. 새 connection 도 `session-store.ts` 의 `openDb()` 를 재사용 (refactor 필요 — `openDb` 가 매번 `DatabaseSync` 생성하므로 `ensureSchema()` 만 한 번 부르도록 wrap).
- 근거: backfill 시 events ↔ messages_fts JOIN (예: surface 이후 transcript 내 키워드 매치) 가 필요. 분리하면 ATTACH 또는 dual-open 으로 복잡도 증가.
- 트레이드오프: DB 파일 비대화. 현 sessions.db 가 어느 정도 크기일지 미측정 — Open Decision §11 (Q1).

### 3.3 동시성 정책
- **WAL 모드 활성화** (`PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA busy_timeout=1000;`) — 다중 hook (`solution-injector`, `post-tool-use`, `compound-extractor`) 가 동시 write 가능.
- `node:sqlite` 는 동기 API — hook 별 own connection, open→ensure schema→insert→close 짧은 트랜잭션.
- 첫 부트 시 `schema_version` row check → 없으면 schema 생성 + WAL 설정. `session-store.ts:openDb` 에 schema 부트스트랩 통합.
- **race 시나리오**: `solution-injector` 가 surfaced emit + `post-tool-use` 가 tool-pattern emit 동시 발생 → SQLite WAL + busy_timeout 1s 로 안전 (현 `rate-limiter.ts` 의 `atomicWriteJSON` 패턴보다 우월).

### 3.4 트랜잭션 단위
- **이벤트당 1 INSERT, 1 commit** — append-only 성격이고 hook latency 가 더 중요 (현 `recordHookTiming` 보면 100ms 이내 budget). batch 최적화는 backfill 도구에서만.

### 3.5 Pruning / Retention
- **기본 180일 보존**. 그 이상은 dead 솔루션 events 만 prune (영구 보존 불필요).
- pruning trigger: (a) `state-gc.ts` cron-like 호출 (이미 존재 `src/core/state-gc.ts`), (b) `fgx maintenance --gc` 명시 CLI.
- 대상 query:
```sql
DELETE FROM solution_events
WHERE ts < (strftime('%s','now','-180 days') * 1000)
  AND solution_id IN (SELECT id FROM dead_solutions);
-- dead_solutions: solution-fitness.ts 의 `state='underperform'` 또는 frontmatter status='dead'
```
- VACUUM 은 prune 이후 size > 100MB 일 때만 수동 권장 (`fgx maintenance --vacuum`).

### 3.6 마이그레이션
- `ensureSchemaV1()` 함수가 `openDb` 첫 호출 시 idempotent 실행.
- 향후 v0.5.x 에서 컬럼 추가 시 `schema_version.version` 비교 → ALTER TABLE.
- 롤백 정책: events 테이블만 분리되어 있으므로 `DROP TABLE solution_events` 1줄로 v0.4.8 호환 회귀 가능.

## 4. Multi-signal acted_on heuristic

### 4.1 Signal 정의 + 가중치

| Signal | 정의 | 추출 위치 (file:line) | 신뢰도 | 가중치 |
|---|---|---|---|---|
| **prompt-keyword** | surface 후 N=5턴 이내 user prompt 에 솔루션 `tags`/`identifiers` 중 ≥1개 출현 | `solution-injector.ts` UserPromptSubmit 진입 시 transcript scan | medium | **0.20** |
| **tool-pattern** | surface 후 5턴 이내 PostToolUse 에서 솔루션이 권장하는 tool 사용 (예: `tdd` 솔루션 → `vitest` 실행, `early-return` → 함수 편집) | `src/hooks/post-tool-use.ts` 또는 `post-tool-handlers.ts` 신규 emit | high | **0.30** |
| **commit-diff** | session 종료 후 30분 내 git commit 의 diff 에 솔루션 권장 패턴 매칭 (regex/identifier 일치) | `spawn.ts:300` exit hook 의 transcript scan 이후 단계 | high | **0.30** |
| **compound-extract** | session 종료 시 `compound-extractor.ts` 가 추출한 새 솔루션의 `supersedes` 또는 본문이 기존 솔루션 reference | `compound-extractor.ts:runExtraction` 이후 hook | very high | **0.20** |

총합 = 1.0. threshold = **0.5** → `signal_score` 합산 ≥ 0.5 시 단일 `acted_on` 이벤트 INSERT (개별 signal 은 별도로 같은 event_type='acted_on' + signal_source 만 다르게 기록 — 분석 시 source 별 contribution 가시화).

### 4.2 가중치 근거
- `tool-pattern` / `commit-diff` 0.30: 행동 증거가 가장 강함. 두 개 모두 hit 만으로도 threshold 도달.
- `compound-extract` 0.20: very high confidence 지만 발생 빈도가 낮음 (extractor 가 gate 통과해야 emit, 하루 최대 5회 — `compound-extractor.ts:33` `MAX_EXTRACTIONS_PER_DAY=5`).
- `prompt-keyword` 0.20: FP 위험 (사용자가 솔루션과 무관한 키워드 우연 입력) — 단독 hit 으로는 threshold 미달.
- "tool-pattern + prompt-keyword" 조합 (0.5) 도 threshold 도달 — 가장 흔한 acted_on 경로.

### 4.3 FP / FN 시나리오

| Signal | False Positive | False Negative |
|---|---|---|
| prompt-keyword | "tdd" 라는 단어가 일반 대화에서 등장 | 사용자가 한국어로 풀어 말함 ("테스트 먼저") — `term-normalizer.ts` 가 일부 커버 |
| tool-pattern | 솔루션과 무관한 시점에 vitest 실행 | 솔루션 적용했지만 tool 변경 없음 (예: code review 솔루션) |
| commit-diff | 우연한 keyword 매칭 | 30분 초과 후 커밋 / 커밋 안 함 |
| compound-extract | 새 추출이 기존 솔루션을 단순 mention | 사용자가 패턴은 썼지만 추출 trigger 미달 |

→ **가중치 합산 + threshold 0.5** 가 단일 signal FP 를 흡수하는 설계. 단독 hit 으로 acted_on 안 됨.

### 4.4 "N턴" 결정
- **N=5** (surface 이후 5 user prompt 또는 5 minutes, 둘 중 짧은 것).
- 근거: 평균 session 길이 (현 `compound-extractor.ts` 가 10+ 메시지에서 trigger — `spawn.ts:353`) 의 절반. 사용자가 surface 본 직후 ~2-3 prompt 내에 행동하면 강한 신호, 그 이상은 noise.
- 구현: surface 이벤트 row 의 `ts` + `session_id` 로 lookup, 이후 prompt-keyword/tool-pattern scan 시 (`ts - surface.ts < 5min`) AND (`prompt_index - surface.prompt_index ≤ 5`) 조건.

### 4.5 가중치 튜닝 방법론
- **수동 + fixture-driven A/B** (자동 학습은 v0.5.0 비목표).
- Ground truth source: backfill 한 누적 데이터에서 다음을 positive 로 라벨링
  - compound-extractor 가 추출한 새 솔루션의 `supersedes` chain 에 reference 된 기존 솔루션
  - solution-outcomes.jsonl 의 `outcome='accept'` 이벤트
- A/B: 가중치 (0.20/0.30/0.30/0.20) vs (0.25/0.25/0.30/0.20) vs (0.15/0.35/0.35/0.15) 등 3-5가지를 fixture 에 돌려 precision/recall 계산.
- 1차 release 는 §4.1 의 (0.20/0.30/0.30/0.20) 으로 고정. v0.5.1 에서 fixture 결과로 재조정.

### 4.6 backfill 로 검증 가능한 ground truth
- ✅ 가능: `compound-extractor` 가 새 솔루션의 본문에 기존 솔루션 `name`/`identifier` 를 mention 한 경우 → 그 기존 솔루션은 "사용됨" positive.
- ✅ 가능: `solution-outcomes/{sid}.jsonl` 의 `outcome='accept'` → 약한 positive (silence-consent 모델이므로 noise 있음).
- ✅ 가능: `compound-usage.jsonl` 의 `recordUsage` — MCP `compound-read` 호출 = 사용자가 명시적으로 본문 읽음. 강한 positive.
- ❌ 한계: "조용히 적용만 하고 commit 안 한" 경우는 ground truth 부재.

## 5. Hit-rate / Lifecycle 분류

### 5.1 SQL View

```sql
CREATE VIEW IF NOT EXISTS v_solution_hit_rate AS
SELECT
  solution_id,
  -- 30d
  SUM(CASE WHEN event_type='matched'  AND ts > (strftime('%s','now','-30 days')*1000) THEN 1 ELSE 0 END) AS matched_30d,
  SUM(CASE WHEN event_type='surfaced' AND ts > (strftime('%s','now','-30 days')*1000) THEN 1 ELSE 0 END) AS surfaced_30d,
  SUM(CASE WHEN event_type='acted_on' AND ts > (strftime('%s','now','-30 days')*1000) THEN 1 ELSE 0 END) AS acted_30d,
  -- 90d
  SUM(CASE WHEN event_type='matched'  AND ts > (strftime('%s','now','-90 days')*1000) THEN 1 ELSE 0 END) AS matched_90d,
  SUM(CASE WHEN event_type='surfaced' AND ts > (strftime('%s','now','-90 days')*1000) THEN 1 ELSE 0 END) AS surfaced_90d,
  SUM(CASE WHEN event_type='acted_on' AND ts > (strftime('%s','now','-90 days')*1000) THEN 1 ELSE 0 END) AS acted_90d,
  -- 180d
  SUM(CASE WHEN event_type='matched'  AND ts > (strftime('%s','now','-180 days')*1000) THEN 1 ELSE 0 END) AS matched_180d,
  SUM(CASE WHEN event_type='surfaced' AND ts > (strftime('%s','now','-180 days')*1000) THEN 1 ELSE 0 END) AS surfaced_180d,
  SUM(CASE WHEN event_type='acted_on' AND ts > (strftime('%s','now','-180 days')*1000) THEN 1 ELSE 0 END) AS acted_180d,
  MAX(ts) AS last_event_ts
FROM solution_events
WHERE event_type IN ('matched','surfaced','acted_on')
  AND signal_source NOT LIKE 'duplicate-%'
GROUP BY solution_id;
```

### 5.2 Lifecycle 분류 (90일 기준)

| Class | 조건 | 근거 |
|---|---|---|
| 🔥 **hot** | `acted_90d ≥ 3` AND `surfaced→acted_on rate ≥ 0.4` | 자주 surface 되고 실제로 자주 적용됨 |
| 🟡 **warm** | `surfaced_90d ≥ 3` AND `acted_90d ≥ 1` (또는 hot 미달) | 적용은 가끔 되지만 살아있음 |
| 🥶 **cold** | `matched_90d ≥ 1` AND `surfaced_90d == 0` | 매칭은 되지만 게이트 통과 못함 (relevance 낮음 등) |
| 💀 **dead** | `matched_180d == 0` AND solution age > 30d | 6개월간 단 한 번도 매칭 안 됨 |
| 🌱 **new** | solution age ≤ 30d | 분류 보류 (cold-start) |

임계값 근거:
- `acted_90d ≥ 3`: 분기마다 1번 미만이면 hot 이라 부르기 약함. 3회면 월 1회 페이스.
- `surfaced→acted_on rate ≥ 0.4`: 40% 적용률 — 현 `MIN_INJECT_RELEVANCE` 통과 후 합리적 기대치. 50% 는 너무 높음 (사용자가 매번 솔루션 따르지 않음).
- `matched_180d==0` AND age>30d: starter-pack 초기 cold-start 보호 + 충분한 노출 기회 후 판단.

### 5.3 hit-rate 통합 점수 (`fgx status` 표시용)
```
hit_rate = acted_90d / max(surfaced_90d, 1)
```
- 0.0~1.0 범위, surface 0 이면 hit_rate 정의 안 함 (분류는 cold/new 처리).

## 6. fgx status 대시보드

### 7.3 emit 구현 패턴 (prompt-keyword 예시)

```typescript
// solution-injector.ts main() 진입부
async function detectActOnFromPriorSurface(sessionId: string, promptLower: string): Promise<void> {
  const recentSurfaces = querySurfacedWithin(sessionId, /*minutes*/ 5);
  for (const surf of recentSurfaces) {
    const sol = loadSolutionById(surf.solution_id); // cached
    if (!sol) continue;
    const hit = sol.tags.some(t => promptLower.includes(t.toLowerCase()))
             || sol.identifiers.some(id => promptLower.includes(id.toLowerCase()));
    if (!hit) continue;
    emitSolutionEvent({
      sessionId,
      solutionId: surf.solution_id,
      eventType: 'acted_on',
      signalSource: 'prompt-keyword',
      signalScore: 0.20,
      meta: { matched_via: 'tag-or-identifier', surface_ts: surf.ts },
    });
  }
}
```

### 7.4 중복 방지
- 동일 (session_id, solution_id, signal_source) 가 5분 window 내 중복 emit 되면 무시 — `signalSource` prefix `duplicate-` 로 marker only (분석 시 노이즈 제거).
- Implementation: `emitSolutionEvent` 내부에서 직전 1건 lookup 후 dedup.

### 8.2 백필 스크립트 위치 + 절차

신규: `src/core/observability-backfill.ts` + `fgx maintenance --backfill` CLI.

**Phase A: 결정론적 신호 backfill (1차 patch)**
1. `match-eval-log.jsonl` 전체 라인 read → 각 line 의 `candidates[]` 의 top-N 을 `matched` events 로 INSERT (signal_score = relevance, signal_source='matcher', meta={matchedTerms})
2. `implicit-feedback.jsonl` 의 `type='recommendation_surfaced'` → `surfaced` event (signal_source='hook-prepend-backfill', signal_score=match_score)
3. `compound-usage.jsonl` 의 각 entry → `acted_on` event (signal_source='mcp-read', signal_score=0.30 — `tool-pattern` 가중치 적용. very high confidence 지만 §4.1 정의에 'mcp-read' 별도 source 추가 검토)
4. `outcomes/<sid>.jsonl` 의 `outcome='accept'` → `acted_on` event (signal_source='outcome-accept', signal_score=0.15)

**Phase B: 휴리스틱 신호 backfill (2차 patch — Phase A 검증 후)**
5. 각 transcript jsonl scan → surface 시점 (matched 이벤트의 ts) 이후 5min/5턴 window 안에서 prompt 의 키워드 매칭 → `acted_on` (signal_source='prompt-keyword', score=0.20)
6. PostToolUse 패턴 추출 (transcript 의 `tool_use` 블록) → `acted_on` (signal_source='tool-pattern', score=0.30)
7. git log scan (현 cwd 기준만) → commit 시각이 surface 이후 30min 안에 있고 diff 가 solution tag/identifier 매칭 → `acted_on` (signal_source='commit-diff', score=0.30)

**Phase C: aggregation**
8. 각 (session_id, solution_id) 별 signal_score 합산 ≥ 0.5 이면 추가 종합 row INSERT (signal_source='aggregated', event_type='acted_on', score=합산값) — view 가 합산만 카운트하면 individual signal 은 분석용으로만 보존.

### 8.3 예상 데이터량
- match-eval-log: 6.5MB → 평균 200B/line → ~32K lines → 솔루션 top-5 평균 = ~160K matched events
- implicit-feedback: 3.1MB → ~15K lines (positive 비율 ~30% 추정) → ~4.5K surfaced events
- compound-usage: 작은 파일 → ~수백 acted_on events
- transcripts (Phase B): ~50개 세션 × 평균 100메시지 → 추가 ~5K acted_on events
- **총 events 테이블 row**: 약 18만 row 예상. SQLite 에 무난.

### 8.4 검증 절차
1. backfill 완료 후 `fgx status` 출력의 hot/warm/cold/dead 분류 사용자에게 보여줌
2. 사용자가 "이 솔루션은 hot 인 게 맞다 / dead 분류는 의외다" 등 라벨링 → ground truth 라벨 파일 (`~/.forgen/state/lifecycle-truth.json`) 생성
3. 자동: classifier 와 ground truth 비교 precision/recall 출력 → §4.5 가중치 튜닝 입력
4. 목표 — 1차 release 는 precision ≥ 0.7, recall ≥ 0.6 (낮은 bar — fixture 가 작아서)

### 8.5 backfill 실행 안전성
- backfill 은 **events 테이블 비어있을 때만** 자동 실행 (`SELECT COUNT(*) FROM solution_events; > 0` 이면 거부, `--force` flag 필요)
- 진행률: 1K events 마다 stderr 출력
- 실패 시 부분 rollback: `BEGIN TRANSACTION ... COMMIT` 으로 phase 별 묶음

## 10. Risk & Open decisions

### Risks

| Risk | 영향도 | 완화 |
|---|---|---|
| SQLite 동시 write contention | medium | WAL 모드 + busy_timeout 1s + emit 단위 < 1ms |
| 누적 transcript 백필 시 GB 단위 read | medium | Phase B 는 lazy / 사용자 명시 trigger 만 (Phase A 는 < 10MB jsonl 들이라 무난) |
| acted_on 가중치 수동 튜닝 비용 | low | v0.5.0 fixed weights, v0.5.1 fixture-driven 재조정 |
| 사용자 프라이버시 — prompt-keyword 가 meta JSON 에 prompt 텍스트 저장? | high | **저장 안 함**. `meta` 에는 `matched_via`/`surface_ts` 등 메타만. 원본 prompt 는 events 에 절대 미보존 |
| statusline-cli ↔ dashboard 역할 중첩 | low | statusline = 항상 보이는 1-3줄 HUD, dashboard = on-demand 전체 표. usage-telemetry 는 둘 다 read (단일 진실) |
| `node:sqlite` Experimental warning | low | 이미 `session-store.ts:17-27` 에서 suppress 처리됨 |
| schema 마이그레이션 실패 시 v0.4.8 회귀 | low | events 테이블만 추가, 기존 messages/sessions 무수정. 롤백 = DROP TABLE |
| compound-extract signal 빈도 너무 낮음 (하루 5회 cap) | medium | acted_on 가중치 0.20 으로 다른 signal 보완 가능. 단독 결정자 아님 |
| FTS5 + events 같은 DB 에서 VACUUM 시 FTS5 인덱스 비용 | medium | VACUUM 은 수동 명령으로만 (`fgx maintenance --vacuum`), prune 은 DELETE 만 |
| backfill 한 acted_on 가 미래 emit 과 중복 | medium | `signal_source='backfill'` prefix 로 구분, dedup view 에서 제외 가능 |

### Open Decisions (P1 시작 전 결정 권장)

1. **emit 동기/비동기** — `node:sqlite` 는 동기 API. hook latency 영향 < 1ms 예상하지만, 최악의 경우 (WAL checkpoint 중) 100ms 가능. **권장: 동기로 시작, hook-timing.jsonl 로 모니터링 후 worker_thread 분리 검토.**

2. **`acted_on` 의 representation** — 옵션 A: 각 signal 별 row + 합산 view, 옵션 B: 단일 통합 row + meta 안에 signal breakdown. **§4.1 의 설계는 A** (분석 가시성 우월) 지만 row 폭증 우려. → §11 Q.

3. **commit-diff signal 의 multi-repo scope** — 현 cwd 만 scan? 전체 머신 git? → 보안/프라이버시상 **cwd 만**.

4. **`mcp-read` 를 acted_on signal 로 추가?** — §8.2 backfill 단계에서 compound-usage.jsonl 을 acted_on 으로 흡수. 추가 가중치 0.25 정도? **권장: tool-pattern 의 한 종류로 분류** (별도 source name 'mcp-read').

5. **dead 솔루션의 자동 retire** — lifecycle dead 된 솔루션을 자동으로 `status: dead` frontmatter 마킹? **권장: v0.5.0 비목표** — 알림만, retire 는 v0.5.1.
