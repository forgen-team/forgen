### W1-2. 에코 하드닝 — **주력 = count 게이트, regex는 보조** *(Rev 2 구조 전환)*
**파일**: `src/core/config-injector.ts`, `src/core/auto-compound-runner.ts`

**실측 근거**: 2026-07-16 격리된 오염 엔트리 49건(JSON) **전부 observedCount=1**. 즉 count 게이트만으로 기존 오염 100% 차단 — regex 목록 증식(whack-a-mole)은 주력 방어가 될 수 없음.

1. **[주력] count 게이트**: `observedCount >= 2`를 **모든 behavior kind에 적용** (preference/thinking/workflow 전부 — 오염 다수가 workflow kind였음). 진짜 사용자 패턴은 재관찰되고, Claude-voice 에코는 같은 문구로 재발하지 않는다.
2. **[보조] 패턴 추가** (`SELF_REFERENTIAL_PATTERNS` — defense-in-depth로만): EN assistant-voice 선두(`I |I'|Let me|Understood…`), 이모지 선두, KO 나레이션 선두(`이해했습니다|확인하겠|현재 상황|준비 완료|안녕하세요…`), 질문-회신형. 기존 H-2 오탐 이력 있으니 앵커(`^`) 엄수.
3. **캡처 사이드 동일 적용**: auto-compound-runner의 behavior persist 직전 판정 — 렌더에서 거르기 전에 저장 자체를 차단.
4. **Tests**: 격리 49건 전수 → 필터됨(count 게이트에서). 유지한 10건 디렉티브 → count 게이트 통과 시나리오(2회 관찰 시뮬). regex 오탐 경계: "이해했습니다"로 시작하는 진짜 사용자 디렉티브가 2회 관찰되면 통과해야 함.

### W3-1. F2 Injection ROI 루프 *(Rev 2 설계 교체 — 기존 quarantine 재사용 불가 확정)*
**실상**: `solution-quarantine.ts`의 `QuarantineEntry = {path, at, errors[]}`는 **frontmatter 파스 에러 기반 파일 격리** — ROI 격리와 저장 모델·회복 시맨틱이 전부 다름 (`pruneQuarantine()`은 파스 성공 시 즉시 해제 → ROI 엔트리는 frontmatter가 유효하니 즉시 풀려버림). `ranking-pipeline.ts`는 순수 함수 모듈("single source of truth for ranking") — observability 의존성 주입은 격리 보장 파괴.

**교체 설계**:
1. **전용 저장소 신설**: `~/.forgen/state/roi-demotions.json` — `{solutionId, demotedAt, windowCount, reason: "low-roi", surfaced, actedOn}`.
2. **판정 위치**: `ranking-pipeline.ts`는 순수 유지. demotion 적용은 `solution-matcher.ts`의 매치 결과 후처리 래퍼에서 (`rankCandidates()` 결과에 ×0.5) — 판정 갱신은 auto-compound 세션 종료 시 `queryHitRate()` 조회로.
3. **임계값** *(Rev 2 하향)*: `surfaced >= 3 && acted_on/surfaced < 0.1` (90d) — 단독 사용자 규모에서 5는 도달 불가로 dead code화. `config.json`으로 조정 가능.
4. **회복**: acted_on 이벤트 발생 시 demotion 즉시 해제. 2회 연속 윈도 강등 유지 → `status: quarantined-roi` (기존 파일-격리와 별개 필드).
5. dashboard "surfaced-but-ignored" top-5 패널.
6. **Tests**: synthetic solution_events → 강등/회복/2윈도 격리/신규 솔루션 유예/임계 경계.
