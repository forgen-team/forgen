# Forgen — 포지셔닝 & 셀링 전략

> **⚠ 2026-07-16 애든덤 (ADR-010 W3-2)** — 아래 본문은 2026-04 기준. 이후 변한 것:
>
> 1. **"학습 루프"는 더 이상 unique 하지 않다.** ECC 가 신뢰도 스코어·진화·프루닝 학습
>    루프를 셀링에 추가했고, 업계 전반(Anthropic 2026 Agentic Coding Trends)이
>    persistent memory + correction 개인화로 수렴했다. forgen 의 차별화는
>    **"학습한다"가 아니라 "학습을 증명한다"** — δ 측정(forgen-eval), acted-on
>    텔레메트리(ROI 루프), calibrate. **코딩 결과물에 대한 도구 기여도(tool-lift)를 judged eval로 재고 공개하는 곳은 없다.**
> 2. **enforcement 는 프론티어 모델이 흡수했다.** v0.4.11 실측: opus-4.8 에서
>    완료 가드 blocks=0 (easy/hard 양쪽), δ 는 100% injection 기여. §3.3 의
>    "6개 안전 가드레일" 중 완료-검증 가드는 opus-4.8/sonnet-5 에서 거의 발화하지
>    않는다 — 모델이 정직해졌다. 그 위에서의 가치는 recall·개인화·측정이며,
>    이를 정직 한계로 공개한다 (§5 스타일). 결정적 가드(secret/db)는 유효.
> 3. **README 히어로 교체("학습을 증명한다")는 W4 재캘리브레이션(R2) 수치 확보
>    후에만** — 수치 없는 "증명" 주장은 자기모순 (실행계획 W3-2, Rev 2).
>
> 경쟁 수치·기능 서술의 출처: `reports/competitive/oss-comparison-2026-07-20.md`
> (2026-07-20 gh api + repo 직접 fetch 실측. ECC=affaan-m/ECC 리네임, OMO=oh-my-openagent
> 리네임·멀티하네스 피벗 확정, ECC 학습루프는 "셀링"이 아니라 **출시 완료**).
> 4. native 흡수 대응(경계 재정의)은 ADR-010 참조 — doctor/usage/권한 중재에서
>    물러나고 moat 4개(correction→profile, 증거 게이팅 정책, compound recall+ROI,
>    multi-host)에 재집중.
>
> **⚠ 2026-07-20 리프레이밍 (δ 히어로 은퇴) — 이 절이 위 §1·§3을 대체한다.**
> R2 저지 재설계(브랜치→PR #73 머지)에서 opus48-hard "거짓완료 압박" 케이스군에
> **측정 가능한 behavioral δ가 없음**이 확정됐다 (전 arm 0/6 노골적 거짓완료,
> blocks=0의 행동확증. 상세 `docs/release/v0.5.0-recalibration.md`). 따라서:
> - **"학습을 증명한다(δ>0)" 히어로는 은퇴한다.** 이 데이터로는 증명할 δ가 없다.
>   숫자 히어로를 억지로 세우지 않는다 — 그게 forgen 이 팔려는 정직성 그 자체다.
> - **"완료를 증명하게 한다(완료가드)" 히어로도 은퇴한다.** 프론티어 모델은 이미
>   정직해서 완료가드가 거의 발화하지 않는다(blocks=0). 이걸 히어로로 두면 약점.
> - **새 히어로 = "당신에게 맞춰지고, 쓸수록 관련 지식을 되살리는 Claude — 그리고
>   자기 코딩-output tool-lift 를 judged eval 로 재고 null 까지 공개하는 유일한
>   도구."** ("효과를 측정하는 유일한 도구"로 넓히지 말 것 — claude-flow 는 인프라
>   벤치를, ECC 는 eval-harness 를 갖는다. 정밀 wedge 는 *코딩 결과물 tool-lift 를
>   judged eval 로 재고 null 까지 공개*하는 것이다.) 제품은
>   개인화(correction→profile)+recall(compound injection)+ROI 강등+결정적 가드,
>   신뢰 wedge 는 **측정하고 정직 공시하는 규율**(δ 숫자가 아니라). 아래 §2·§3 는
>   이 축으로 재작성됨.

## 2. Forgen 포지셔닝

### 2.1 한 줄 포지셔닝

```
다른 도구는 Claude를 더 강하게 만든다.
Forgen은 Claude를 당신에게 맞춘다 — 그리고 자기 코딩-output tool-lift 를
judged eval 로 재고, 효과가 없으면 null 까지 공개하는 유일한 도구다.
```

> 정직성 노트 (2026-07-20): 이전 문구 "쓸수록, 더 정확하게"는 *정량적 개선*을
> 암시했으나 우리 하드 케이스군에는 측정 가능한 behavioral δ가 없었다(위 애든덤).
> 그래서 개선을 *수치로* 약속하지 않는다. 대신 개인화·recall 이라는 *메커니즘*과,
> 효과를 재고 null 까지 공개하는 *규율*을 판다. 이게 경쟁자와 다른 지점이다 —
> 모두가 "Claude 를 낫게 한다"고 (검증 없이) 주장하고, 아무도 자기 효과를 재서
> 공개하지 않는다.

### 2.2 핵심 프레이밍

**경쟁자들의 프레이밍:**
```
ECC:        "Claude + 자동 학습 루프 + 신뢰도 스코어 + cross-user 공유 = 에이전트 통합 플랫폼"
gstack:     "Claude + 23개 역할 = 소프트웨어 팩토리"
OMC:        "Claude + 19개 에이전트 = 스테로이드"
OMO:        "Claude + OpenCode + Codex + 멀티하네스 = 통합 에이전트 오케스트레이터"
claude-mem: "Claude + 메모리 압축·회상 = 기억하는 AI"
claude-flow: "Claude + 벡터 메모리 + 라우팅 최적화 = 에이전트 하이퍼바이저"
```

**Forgen의 프레이밍:**
```
"Claude + 당신의 교정·솔루션 = 당신에게 맞춰지고, 자기 효과를 정직하게 재는 Claude"
```

다른 도구들은 Claude에 **기능**을 더한다.
Forgen은 Claude에 **당신**을 더하고 — 그 효과를 **측정**한다.

효과 측정이 왜 프레이밍의 일부인가: 경쟁자는 전부 "Claude 를 낫게 한다"를
*검증 없이* 주장한다. forgen 은 judged eval 로 재고, 이번처럼 "이 케이스군엔
측정 가능한 효과가 없다"는 null 도 공개한다(`docs/release/v0.5.0-recalibration.md`).
불편한 진실을 말하는 도구가, 편한 주장만 하는 도구보다 신뢰된다.

### 2.3 적이 아닌 것

Forgen은 ECC/OMC/gstack과 **경쟁하지 않는다**.
오히려 **함께 쓸 수 있다**.

```
gstack의 /qa로 브라우저 테스트하면서
+ forgen의 compound가 "지난번에 이 페이지에서 발견한 버그" 알려줌

OMC의 ralph로 끝까지 돌리면서
+ forgen의 solution-injector가 "이전에 같은 에러 해결한 방법" 주입

이것이 가능한 이유: forgen은 hooks + MCP 레이어에서 동작.
다른 도구의 스킬/에이전트와 충돌하지 않음.
```

이 "함께 쓸 수 있다" 포지셔닝은:
- 기존 도구 사용자를 적으로 만들지 않음
- "추가로 설치하면 더 좋아진다" → 진입 장벽 낮춤
- 경쟁이 아닌 보완 관계 → 커뮤니티 반감 없음

## 3. 히어로 메시지 설계

### 3.1 README 히어로 섹션

```markdown
# forgen

**Claude를 당신에게 맞추는 하네스.**
**당신의 교정을 기억하고, 지난 해결책을 되살리고 — 그 효과를 정직하게 잽니다.**

---

forgen이 실제로 하는 일 (검증된 메커니즘):

- **교정하면 기억한다** — "그렇게 말고 이렇게"가 지속 규칙이 되어 다음 세션에 적용.
- **해결하면 되살린다** — 지난 세션에서 통한 솔루션이 관련 맥락에 자동 주입.
- **낡으면 강등한다** — 자주 뜨는데 안 쓰이는 솔루션은 ROI 루프가 조용히 격리.
- **위험은 결정적으로 막는다** — secret/DB 가드는 모델 정직성과 무관하게 발화.

그리고 — **자기 효과를 측정한다.** judged eval 로 "이게 실제로 돕는가"를 재고,
답이 "이 케이스엔 측정 가능한 효과 없음"이어도 그대로 공개합니다
(`docs/release/v0.5.0-recalibration.md`). 편한 주장 대신 잰 진실을 말합니다.

> 정직성: 위 "1회차→10회차 교정 횟수 감소" 같은 개선 곡선은 *예시가 아니라 실측만*
> 싣는다. 우리 하드 케이스군엔 측정 가능한 behavioral δ가 없었으므로(blocks=0),
> 개선을 수치 표로 약속하지 않는다. 메커니즘은 진짜고, 효과 크기는 정직하게 미주장.

## 30초 시작

\```bash
npm i -g @wooojin/forgen
forgen
\```

4개 질문에 답하면 끝. 바로 사용 가능.
기존 Claude Code 위에 설치됩니다. 충돌 없음.
```

### 3.2 "Aha 모먼트" 설계

**1단계 (설치 직후, 1분)**: 4개 온보딩 질문 → 프로필 생성
- "아, 이게 나한테 맞춰주는 거구나"

**2단계 (첫 세션 종료, 30분)**: compound 추출 제안
- "이 세션에서 3개 패턴을 발견했습니다. 저장할까요?"
- "아, 이게 기억해주는 거구나"

**3단계 (두 번째 세션, 다음 날)**: compound 자동 주입 ← **진짜 Aha**
- "이전 세션에서 학습한 패턴: prisma-upsert-pattern"
- "아, 어제 내가 한 게 오늘 도움이 되네"

경쟁자의 Aha는 "설치 직후" 발생 (ralph, ultrawork 실행).
Forgen의 Aha는 "두 번째 세션"에서 발생.

**문제**: 2단계까지 가기 전에 이탈할 수 있다.
**해결**: 1단계에서 즉시 가치를 보여줘야 한다.

### 3.3 1단계 즉시 가치 — "안전 가드레일"

설치 직후 compound 학습 효과를 기다리지 않아도 얻는 가치:

```
forgen을 설치하면 즉시 활성화되는 것:
├── secret-filter    — API 키 유출 자동 차단
├── slop-detector    — AI가 남긴 TODO, as any 감지
├── db-guard         — DROP TABLE, WHERE 없는 DELETE 차단
├── rate-limiter     — MCP 과도 호출 방지
├── context-guard    — context limit 접근 시 자동 대응
└── session-recovery — 세션 중단 시 자동 복구
```

**메시지**: "설치만 하면 6개 안전망이 즉시 작동합니다. 학습 효과는 보너스."

이것으로 1단계 Aha를 만든다:
- 설치 → 코딩 중 `rm -rf` 입력 → forgen이 차단 → "이거 좋은데?"
- 설치 → Edit에서 API 키 노출 → forgen이 경고 → "이거 필요했어"

## 4. 셀링 포인트 3개 (feature가 아닌 story)

> 프레이밍 규칙 (2026-07-20): 아래 스토리는 *메커니즘 예시*다 — recall/개인화가
> "이렇게 동작한다"를 보여준다. 등장하는 시간 수치("3시간 절약" 등)는 서사적
> 예시이지 측정된 효과 크기가 아니다(honest-null). "평균 N% 빨라진다"처럼
> 일반화·정량 주장으로 읽지 말 것.

### Story 1: "같은 실수를 두 번 하지 않는 Claude"

```
Day 1: PG 연동에서 웹훅 HMAC 검증을 빠뜨려서 오래 삽질.
       → forgen이 "tosspayments-webhook-hmac" 패턴으로 기록.

Day 15: 구독 결제 기능 추가.
        → forgen이 자동으로 "이전에 웹훅 HMAC 검증 빠뜨린 적 있음" 경고.
        → 처음부터 그 함정을 피함 (예시 — 절약량은 상황마다 다름).

Day 30: 다른 프로젝트에서 Stripe 연동.
        → scope:me 솔루션이 프로젝트를 넘어 자동 주입.
        → "결제 연동 시 웹훅 검증 필수" 패턴 적용.
```

### Story 2: "교정하면 기억하는 Claude"

```
Week 1: "너무 많이 바꿨어. 최소한만 수정해."
        → forgen이 correction-record로 기록.
        → judgment_philosophy 축이 "최소변경형"으로 조정.

Week 2: 같은 상황에서 Claude가 최소 변경만 제안.

## 12. 요약 — 셀링 공식

> ⚠ 2026-07-20 정정: 아래 공식 중 PROOF/결론 줄은 은퇴한 δ-히어로였다 —
> honest-null(측정 가능한 behavioral δ 없음) 이후 "정량 개선을 증명한다"는 셀링
> 축은 폐기했다. PROOF 는 이제 *숫자*가 아니라 *규율*(측정하고 null 까지 공개)이다.

```

WHO:   3개월+ 같은 프로젝트에서 일하는 개발자
WHY:   "쓸수록 나에게 맞춰지는 Claude"
HOW:   compound recall + 4축 프로필 + evidence 교정 + ROI 강등
PROOF: 자기 효과를 judged eval 로 재고 null 까지 공개하는 규율
       (정량 개선 대시보드 아님 — 우리 하드 케이스엔 측정 δ 없음)
HOOK:  설치 즉시 결정적 안전 가드레일 (즉시 가치)
MOAT:  학습 축적 → 사용할수록 전환 비용 증가 → lock-in

  ✅ 10개 킬러 스킬 (deep-interview 266줄 ~ docker 146줄)

  ECC:     "우리도 학습해요" ← 학습은 이제 다 함(수렴) X
